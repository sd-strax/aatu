# Knowledge Service — Spec

## Project context

The agent loop reasons over evidence and proposes actions; the quality of that reasoning depends on what the LLM knows about *this org*. Generic LLMs don't know the org's tribal procedures, recent incidents, or the patterns the senior analysts have learned the hard way. The knowledge service is where that institutional context lives — retrieved at reasoning time, cited in rationale, audit-linked to the Interpretation that consulted it.

This spec replaces what a SOAR-era design would have called "playbooks." Playbooks codify *control flow*; the knowledge service captures *judgment*. The LLM still chooses which capabilities to invoke; the knowledge service tells it what this org thinks about cases like this. The reasoning thread shows how the SOP applied, not which step in a script ran.

Two corpora at v0+:
- **SOPs** — institutional standing knowledge. Versioned, signed prose authored by senior analysts. Retrieved by relevance to investigation context.
- **Concluded investigations** — institutional case knowledge. Auto-extracted summaries of the tenant's own past work. Retrieved by similarity to current investigation seed/state.

Both expose the same retrieval surface to the agent loop. Both are per-tenant. Both audit-link into the reasoning thread. Both inform the LLM as context, not as control.

## Thread scope

- The two corpora — schemas, lifecycle, governance
- Retrieval API and ranking
- Agent loop integration: implicit retrieval and explicit tool calls
- Audit linkage to Interpretation events (Layer A and Layer B)
- Embedding model choices: bundled local model and BYOK provider embeddings
- Per-tenant isolation
- Authoring and signoff governance
- The post-conclusion summary extraction workflow's entry point (full pipeline detailed in 07)

## Out of scope

- The capability layer (03-capability-layer.md) — knowledge service is a sibling, not a member; capability outputs are world-facing observations, knowledge outputs are internal context
- Component topology and deployment (05-component-architecture.md) — the service is a known-shape component there
- Post-conclusion outputs broadly (07-post-conclusion-outputs.md) — IOC extraction, ticketing handoff, document generation, etc.
- Detection authoring as a feature (deferred to v2+)
- Cross-tenant federation of SOPs or summaries (deferred indefinitely; per-tenant isolation is a hard property)

## 1. Architectural commitments

**1. Two corpora, one retrieval surface.** SOPs and concluded-investigation summaries are different data — institutional standing knowledge vs. institutional case knowledge — but the agent loop consumes both as ranked context. The retrieval API is uniform: scope, query, return ranked results with rationale. Future corpora (e.g., a tenant-curated threat actor profile library) extend the same surface without changing the agent loop.

**2. Per-tenant by construction.** Both corpora are tenant-scoped. SOPs authored in tenant A are invisible to tenant B. Investigation summaries from tenant A are invisible to tenant B. Cross-tenant retrieval is not expressible. Per the per-tenant namespace UUID rule (01-domain-model.md §3, §5), this is structural, not just policy.

**3. Knowledge is context, not control.** Retrieved knowledge enters the LLM's reasoning as context — system prompt material or tool-call results. It never becomes graph content. SOPs and summaries are not STIX-shaped objects in the interpretation layer; they are sibling stores. The Interpretation that *consulted* knowledge cites it via Layer A reference fields, but the knowledge itself does not enter the aggregate.

**4. Audit-linked, tamper-evident.** Every Interpretation that consulted knowledge records `consulted_sops` and `consulted_similar_investigations` references on its event payload, with content hashes. Layer B side stores hold the actual retrieved snippets keyed by those hashes. If an SOP or summary is later modified, the hash on the Interpretation surfaces the change — preserving the property that "what did the AI see when it cited this SOP?" is answerable for any past Interpretation.

**5. Versioned and signed.** SOPs carry version, author, signoff. Investigation summaries carry the version of the extraction workflow that produced them and the model that generated any LLM-derived parts. Retrieval surfaces both the content and the version that was active at retrieval time.

**6. Human-authored, AI-assisted, never AI-decided.** SOPs are written by senior analysts. Candidate SOPs may be generated by the post-conclusion pipeline (07), but they enter as drafts requiring `sop_signer` review before publication. The system never silently inserts judgment into the corpus the LLM treats as authoritative.

**7. Embedding-agnostic storage.** The storage layer abstracts over embedding model choices. v0+ default is pgvector with a bundled small ONNX model; tenants can switch to BYOK provider embeddings (the same provider used for LLM inference) for higher retrieval quality. Switching models triggers re-embedding of the corpus; embeddings are versioned by the model that produced them.

## 2. Corpus 1 — SOPs

### 2.1 Shape

```
SOP

  id                  sop--<uuid v4 within tenant namespace>
  tenant_id           UUID
  title               string
  body                markdown
  status              DRAFT | IN_REVIEW | PUBLISHED | RETIRED
  version             string (semver: 1.0.0, 1.1.0, 2.0.0)
  applies_to          {
                        entity_types?      list<string>   # e.g., ["x-host", "user-account"]
                        techniques?        list<string>   # MITRE ATT&CK technique IDs
                        threat_categories? list<string>   # e.g., ["ransomware", "bec"]
                        severity_min?      "LOW" | "MEDIUM" | "HIGH" | "CRITICAL"
                      }
  authored_by         user_id (must hold sop_author role)
  signed_off_by       list<user_id> (each must hold sop_signer role)
  effective_from      timestamp
  effective_until     optional timestamp
  content_hash        sha256 of canonical (title + body + applies_to)
  embedding_refs      list<{model, version, vector_id}>
  supersedes_ref      optional sop id (when this is a new version replacing an older one)
  created, updated    timestamps
```

**`body` content conventions.** SOPs are prose, not procedure code. They state priorities, exceptions, communication norms, references to past incidents, and judgment calls. They do *not* enumerate executable steps. The system never parses the body into a workflow; the LLM reads it as context. Authoring guidelines (a separate document, not in this spec) recommend short headers, explicit exception cases, and citations of past investigations or external references where applicable.

**Identity is UUIDv4 within the tenant namespace, not deterministic.** SOPs are documents, not observable facts; deterministic identity rules don't apply. Two SOPs with identical body in different tenants are deliberately distinct entities.

### 2.2 Lifecycle

```
DRAFT       →  IN_REVIEW    (sop_author submits)
IN_REVIEW   →  DRAFT        (sop_signer requests changes)
IN_REVIEW   →  PUBLISHED    (sop_signer signs off; effective_from set)
PUBLISHED   →  RETIRED      (sop_signer retires; effective_until set)
RETIRED     →  (terminal)
```

Edits to a PUBLISHED SOP create a new version (incrementing the version field) in DRAFT, supersedes the prior version on publication. The prior version stays accessible for retrieval audit (cited Interpretations from before the edit can still resolve the SOP they consulted) but is no longer returned by retrieval queries by default.

**Signoff.** PUBLISHED SOPs require at least one user holding `sop_signer` role to sign off. Tenants may configure higher requirements (e.g., two `sop_signer`s for SOPs tagged with severity_min = CRITICAL). The signoff record is part of the immutable audit trail of the SOP.

### 2.3 Storage

CRUD table per tenant in `aatu_knowledge.sops`. Embeddings live in `aatu_knowledge.sop_embeddings` keyed by `(sop_id, sop_version, model, model_version)`. Body content is stored as text on the row; large bodies (>1MB, edge case) overflow to side store with a content-hash reference, same pattern as Layer B for transcripts.

### 2.4 Authoring

Editing happens through aatu's tenant-admin web UI or, for solo subscribers, through the VS Code extension's SOP editor. The flow:

1. `sop_author` creates DRAFT, edits body, sets `applies_to` metadata
2. Submits for review → IN_REVIEW
3. `sop_signer` reviews, requests changes (back to DRAFT) or signs off (→ PUBLISHED)
4. On PUBLISHED: embeddings generated, retrieval index updated

Tenants may configure a "quick path" where `sop_author` and `sop_signer` are the same user for low-risk SOPs (e.g., severity_min ≤ MEDIUM) — relevant for solo subscribers and small teams. Higher-risk SOPs require distinct author and signer.

### 2.5 Solo subscribers

Solo subscribers hold both `sop_author` and `sop_signer` roles in their personal tenant. They can author SOPs for their own use without signoff ceremony. The system records their dual role on the SOP — when the SOP lifts to a SaaS tenant via the lift path (05 §9), the receiving tenant's signoff requirements may demand re-review before the SOP is published in the new tenant.

## 3. Corpus 2 — concluded-investigation summaries

### 3.1 Shape

```
InvestigationSummary

  id                  summary--<uuid v4 within tenant namespace>
  tenant_id           UUID
  investigation_ref   grouping--<uuid> (the source investigation)
  conclusion_at       timestamp (when the source investigation concluded)
  summary_text        string (LLM-generated narrative summary; bounded ~2000 chars)
  structured          {
                        seed_kind          "alert" | "entity" | "question"
                        seed_value         string (alert id, entity ref, hypothesis statement)
                        primary_entities   list<{entity_ref, type, role}>
                        techniques         list<string>   # MITRE ATT&CK technique IDs
                        primary_indicators list<{indicator_ref, value, type}>
                        actions_taken      list<{action_type, target_count, outcome}>
                        conclusion_outcome "succeeded" | "failed" | "abandoned" | "inconclusive"
                        duration_hours     number
                      }
  extractor_version   string (version of the extraction workflow)
  generator_model     string (LLM and version that produced summary_text)
  content_hash        sha256 of (summary_text + structured)
  embedding_refs      list<{model, version, vector_id}>
  created             timestamp
```

The summary is *derived* from the source investigation, not authored directly. It exists to support similarity retrieval and post-conclusion analytics. Edits to the summary trigger re-extraction; manual override is permitted but flagged in audit.

### 3.2 Extraction workflow

Triggered by `InvestigationConcluded` events. Implemented as the `SummarizeForKnowledgeIndex(grouping_id)` Temporal workflow (05 §3.3, also referenced in 07).

Steps:
1. Load the concluded investigation's projection (`investigation_current`) and reasoning thread (`investigation_thread`)
2. Extract structured fields directly from the aggregate:
   - `seed_kind`, `seed_value` from the immutable seed
   - `primary_entities` from Grouping `object_refs`, ranked by edge density and member-of recency
   - `techniques` from `x-hypothesis.labels` and from Indicators' `indicator_types`
   - `primary_indicators` from Indicators with `produced-by` edges to Interpretations in the thread
   - `actions_taken` from `x-action` lifecycle events
   - `conclusion_outcome` from the thread's `conclusion` Interpretation rationale, with LLM disambiguation if not classifiable from structure alone
   - `duration_hours` from seed timestamp to conclusion timestamp
3. Generate `summary_text` via LLM call: prompt with the structured fields and a curated subset of high-signal Interpretations from the thread; LLM produces the narrative
4. Compute embeddings on `summary_text + structured` representation
5. Write the summary row, embeddings, and audit record (which extractor version, which LLM, content hash)

The extraction's LLM call uses the BYOK key for solo, the tenant-configured key for SaaS. If no LLM key is configured, structured-only summaries are produced (no `summary_text`); retrieval quality drops but functionality continues.

### 3.3 In-flight investigation indexing

Optional. When enabled, active investigations are indexed at coarser granularity (every N hours, or on milestone events like new hypothesis SUPPORTED). The retrieval API distinguishes concluded (canonical) from in-flight (provisional) results; the agent loop typically prefers concluded. Useful for "is anyone else right now investigating something that overlaps?" in multi-analyst SaaS, less so in solo.

### 3.4 Storage

CRUD table `aatu_knowledge.investigation_summaries`. Embeddings in `aatu_knowledge.summary_embeddings`. Summary text fits in the row; it is bounded by design.

### 3.5 Retention

Summaries are retained as long as the source investigation is retained. Archived investigations (Lifecycle = ARCHIVED, 01-domain-model.md) retain their summaries — that is the whole point of the corpus, retrospective context. If a customer requires summary purging for compliance, the lifecycle workflow extends to delete the summary alongside the source data.

## 4. Retrieval API

The agent loop consumes knowledge via two endpoints exposed by the backend:

```
POST /knowledge/recall_sops
  Request:
    scope                {entity_types?, techniques?, threat_categories?, severity_min?}
    query                string (free-form, used for embedding similarity)
    limit                int (default 5)
    include_retired      bool (default false; for audit/research only)
  Response:
    results              list<{sop_id, version, title, excerpt, score,
                               match_rationale, content_hash}>
    coverage             "COMPLETE" | "EMPTY"
    retrieval_at         timestamp

POST /knowledge/recall_similar_investigations
  Request:
    scope                {techniques?, primary_entities?, seed_kind?, seed_value?}
    query                string (typically the current investigation seed or a hypothesis)
    limit                int (default 5)
    include_in_flight    bool (default false; concluded only by default)
  Response:
    results              list<{summary_id, investigation_ref, conclusion_at,
                               outcome, summary_excerpt, score, match_rationale,
                               content_hash}>
    coverage             "COMPLETE" | "EMPTY"
    retrieval_at         timestamp
```

### 4.1 Ranking

For each corpus the ranker combines:
- **Vector similarity** between the query embedding and the corpus embeddings (cosine)
- **Metadata filtering** from the `scope` fields (techniques, entity types, etc.) — applied as a hard filter before ranking
- **Recency boost** for SOPs (newer versions ranked higher) and summaries (recent investigations ranked higher when scores are close)
- **Signoff weight** for SOPs (multiple signers signal higher institutional weight)

The ranker emits a `match_rationale` per result: a short prose explanation of why the result ranked where it did ("matches techniques T1486 and T1078.004 (hard filter); summary embedding similarity 0.82; conclusion outcome was 'succeeded'"). This is presented to the LLM alongside the excerpt, helping it reason about the relevance.

### 4.2 Excerpts

For SOPs: the full body up to a token budget (default 4000 tokens). For longer SOPs, return the most-relevant section identified by sub-document embedding.

For summaries: the `summary_text` plus the `structured` fields. The agent loop can drill down via a follow-up call to fetch the full investigation projection if it wants more context.

### 4.3 Coverage

Per the 03-capability-layer.md coverage convention, retrieval emits a coarse outcome:
- `COMPLETE` — corpus was queried, results returned (possibly empty if no matches)
- `EMPTY` — corpus has no entries matching the scope (e.g., no SOPs tagged for the relevant techniques)

`EMPTY` is meaningful evidence-of-absence: "we have no institutional procedure for this" is a real signal the LLM should weigh.

## 5. Agent loop integration

### 5.1 Implicit retrieval at investigation seed time

When an investigation enters ACTIVE state (or a new turn begins on an active investigation), the agent loop performs an automatic SOP recall against the seed context. The query is constructed from:
- Seed kind and value
- Initial entity types and IDs in the Grouping
- Any techniques surfaced from the seeding alert (if seed is `AlertSeed`)

Returned SOPs go into the system prompt under a clearly-marked section: `# Your organization's relevant SOPs for cases like this`. The LLM consumes them as background context; no tool call needed.

A symmetric implicit recall hits `recall_similar_investigations` with the same scope. The top N similar past concluded investigations enter the system prompt under `# Similar past investigations in your tenant`.

Both implicit retrievals run before the LLM's first inference call on the turn.

### 5.2 Explicit tool calls

The LLM tool catalog includes:

```
recall_sop(query: string, scope?: {...}) -> list<SOPResult>
recall_similar_investigations(query: string, scope?: {...}) -> list<SummaryResult>
```

The LLM invokes these when reasoning surfaces a question implicit retrieval missed. ("This is starting to look like a supply-chain compromise; pull SOPs tagged with `T1195`.") Tool results enter the conversation as tool-result messages.

These are *not* capability-layer verbs. They produce no telemetry, no OcsfEvents, no STIX nodes. They are knowledge-service calls, side-channel from the world-facing tool surface. The architectural separation is deliberate (05 §6): capability layer stays pure I/O for telemetry; knowledge service is internal context retrieval.

### 5.3 System prompt scaffolding

The agent loop's system prompt includes guidance on when to weigh SOP recommendations, how to handle conflicts between SOPs and current evidence, and how to cite SOPs in rationale. Authored once, refined through operation. This sits in the `prompts/` directory of the codebase (per 05 §3.4) and is consumed identically by the extension's TypeScript runtime and any future Go-side agent runtime.

Specifically, the prompt instructs the LLM to:
- Cite SOPs by ID and version in Interpretation rationale when their guidance influenced the decision
- Surface explicit deviations: "SOP X recommends Y but I'm proposing Z because [evidence-specific reason]" — recorded in rationale
- Treat the absence of relevant SOPs as informative ("no SOP applies, reasoning from primitives")
- Cite similar past investigations only when the LLM uses them substantively, not for every recall

## 6. Audit linkage

### 6.1 Layer A — structured references on the Interpretation event

Per the companion edits in 05 §13.1, the Interpretation event payload (02-persistence.md §6) gains optional fields:

```
consulted_sops                       list<{sop_id, version, retrieval_score, used: bool}>
consulted_similar_investigations     list<{summary_id, version, retrieval_score, used: bool}>
```

The `used` flag distinguishes SOPs/summaries the LLM merely retrieved from those it cited in its rationale. Retrieved-but-not-used items are still recorded — they show what context the LLM had access to when deciding.

These fields are queryable. "Show me every Interpretation that cited SOP-RANSOMWARE-CONTAINMENT v3.2" is a single SQL query against the projection. "Show me cases where the LLM had access to SOP X but chose not to cite it" is similarly direct.

### 6.2 Layer B — retrieved snippets in side store

The actual retrieved SOP excerpts and summary snippets land in a Layer B side store (`aatu_main.knowledge_retrievals` table, structurally identical to `ai_tool_calls`). Each row is keyed by content hash; multiple Interpretations citing the same SOP version share the same row.

Tamper evidence: the hash recorded on the Interpretation event verifies the bytes the LLM actually saw at retrieval time. If an SOP is later edited (creating a new version), the original version's content remains retrievable via its hash; the bytes the LLM cited are still inspectable.

### 6.3 Provenance extension

The `producer_version` field (01-domain-model.md PROVENANCE) on Interpretation outputs gains an optional `retrieval_context_hash` component: a hash over the union of `consulted_sops` and `consulted_similar_investigations` for the Interpretation. Re-running an investigation under the same retrieval context (same SOPs, same versions, same summaries available) produces a stable retrieval-context hash, which supports behavioral consistency analysis across runs.

## 7. Embedding model

### 7.1 Bundled local model

The aatu binary ships with a small ONNX embedding model (low hundreds of MB, e.g., a quantized BGE-small or equivalent). Default for solo subscribers. Embedding generation happens in the backend process; no network calls for embedding.

The bundled model is versioned. Updates ship via the CDN. Tenants pin to specific versions; updating triggers re-embedding of the corpus.

### 7.2 BYOK provider embeddings

Tenants can configure embedding to use their LLM provider's embedding endpoint (Anthropic, OpenAI, Cohere, etc.) when higher retrieval quality matters and the BYOK key has access to embeddings. Configured per-tenant; falls back to bundled model if the provider call fails.

Provider embeddings are typically larger (1024+ dimensions) and produce sharper retrieval. Cost: small per-document, low-frequency at v0+ scale (~hundreds of SOPs and summaries per tenant).

### 7.3 Switching models

When a tenant switches embedding models:
1. New embeddings are generated for the entire corpus on the new model
2. Both old and new embedding rows coexist during the migration window
3. Retrieval queries use the new model's embeddings once migration completes
4. Old rows are pruned after a configurable retention window (default 30 days)

The `embedding_refs` field on each corpus row carries multiple embeddings; queries match against the active model's embeddings.

### 7.4 Search-only vs LLM-aware retrieval

The retrieval API as specified is search-only: vector similarity + metadata filtering + ranking heuristics. v2+ may introduce LLM-aware retrieval — a small LLM call to rerank the top-K results based on the query — when retrieval quality at scale demands it. This is an additive enhancement that doesn't change the API contract.

## 8. Per-tenant isolation

Both corpora are tenant-scoped at every level:

- **Storage rows** carry `tenant_id`; queries filter on it; row-level security policies in SaaS provide defense-in-depth.
- **Embeddings** are stored in tenant-partitioned indexes; vector queries cannot cross tenants.
- **Retrieval API** requires JWT authentication; the JWT's `tenant_id` claim scopes all queries. There is no admin-level "search all tenants" affordance for SaaS staff (and aatu staff have no read access to tenant content under any role).
- **Lift path** (05 §9) preserves SOPs and summaries during Sub-path A (tenant of one). For Sub-path B (joining existing tenant), local SOPs do not auto-merge into the shared tenant; the subscriber's local SOPs remain in their personal-scratch tenant unless explicitly imported (with tenant admin signoff).
- **Cross-tenant federation** is not expressible. A future "shared SOP library" (deferred indefinitely) would be a separate corpus with its own publication and signing model, never silent merge.

## 9. Authoring UX (high-level)

This section sketches the analyst-facing surface; full UX is product-side, not in this spec.

### 9.1 SOP editor

- Web (SaaS) and VS Code extension (solo and SaaS) both expose a markdown editor with live preview.
- Metadata fields (`applies_to`, severity, etc.) edited via structured inputs alongside the body.
- Submit-for-review action emits an `IN_REVIEW` transition; the SOP appears in the signer's queue.
- Signers see diff against prior version, can request changes or sign off.
- Embedding regeneration happens automatically on PUBLISHED transition; status indicator shows when retrieval index is current.

### 9.2 Library view

- List of SOPs filterable by `applies_to`, status, author, last-modified.
- Per-SOP detail page shows version history, signoff record, retrieval analytics ("cited in 47 investigations over the last 90 days").
- Retired SOPs accessible but marked.

### 9.3 Candidate-SOP review

The post-conclusion pipeline (07) may generate candidate SOPs from concluded investigations — patterns the system noticed worth codifying. These enter the SOP library as DRAFTs flagged `source: post_conclusion_candidate`, with the source investigation's reference attached. `sop_author` reviews, edits, refines, optionally promotes through the normal lifecycle. The system never silently publishes a candidate.

## 10. v0 / v1 / v2+ staging

| Stage | Capabilities |
|---|---|
| **v0** | SOP CRUD with manual authoring; keyword retrieval (Postgres full-text); no embeddings yet; no investigation summaries (the post-conclusion pipeline doesn't run at v0 because there's nothing to draw from in fixture-only mode). The retrieval API exists but returns keyword matches. |
| **v1** | Embeddings via the bundled local model; investigation summary extraction at conclusion (real investigations now exist); both corpora retrievable via vector similarity. SOP candidate generation begins (rough quality). |
| **v2** | BYOK provider embeddings for tenants that want sharper retrieval; rich `match_rationale` generation; in-flight investigation indexing for multi-analyst SaaS; SOP candidate quality improves with operation. |
| **v3+** | Optional LLM-aware reranking; cross-investigation linkage events feeding the summary corpus with explicit relationships ("this case is a follow-on to that one"); tenant-curated additional corpora (threat actor profiles, custom playbooks-as-prose). |

## 11. Open questions / Deferred to implementation

- **Bundled embedding model choice.** BGE-small, MiniLM, or equivalent. Trade-off: model size in the binary vs retrieval quality. v0 default is configurable; first customer feedback drives the choice.
- **Retrieval token budget.** SOPs can be long; default budget is 4000 tokens but tunable per LLM model context window.
- **Sub-document embedding for long SOPs.** v0+ uses whole-document embedding with section-extraction at retrieval time; v1 may move to sub-document embedding for sharper section-level matches.
- **Match-rationale generation cost.** v0+ produces match rationale via simple template; v2+ may use a small LLM call to generate richer rationale. Balance cost vs interpretability.
- **In-flight investigation indexing freshness.** N-hour granularity vs event-driven re-indexing. Operational tuning, not architecture.
- **Candidate-SOP generator quality.** v1 candidate SOPs may be rough — the goal at v1 is "useful drafts that a human refines," not "production-ready." Quality bar rises as the pipeline matures.
- **Authoring UX for solo subscribers.** Whether VS Code extension is sufficient or a web editor is also needed for solo. Pragmatic call after first real users.
